# CampusTask — 校园任务清单

一个命令行任务管理工具，支持添加任务、查看任务列表、标记完成任务。数据持久化到 `tasks.json`。

## 用户故事与验收标准

以下用户故事基于对 2 名同学的访谈，了解他们在管理课程作业、实验报告和考试复习任务时遇到的问题。

### 用户故事 1

> 作为学生，我希望记录实验报告的截止日期，以免忘记提交。

**验收标准：**
- 执行 `python main.py add "提交数字电路实验报告"` 后，程序输出任务编号和标题
- `tasks.json` 中出现该任务，包含编号、标题、状态（todo）和创建时间

### 用户故事 2

> 作为学生，我希望查看所有待办任务，以便规划复习时间。

**验收标准：**
- 执行 `python main.py list` 后，控制台列出所有任务
- 已完成的显示 ✓，未完成的显示 ○
- 无任务时显示"暂无任务"

### 用户故事 3

> 作为学生，我希望把已完成的任务标记为"完成"，以便跟踪进度。

**验收标准：**
- 执行 `python main.py done 1` 后，编号为 1 的任务状态变为 done
- `tasks.json` 中对应任务状态更新
- 对不存在的编号给出错误提示
- 对已完成的编号提示"已经是完成状态"

### 用户故事 4

> 作为学生，我希望每个任务有唯一编号，以便和同学讨论时快速引用。

**验收标准：**
- 新增任务时编号自动递增
- 删除已有任务不会导致编号重复（编号基于历史最大值 + 1）

### 用户故事 5

> 作为学生，我希望关闭程序后任务不会丢失，下次打开还能看到。

**验收标准：**
- 任务数据存储在 `tasks.json` 文件中
- 程序启动时自动读取已有任务
- 添加/完成任务后立即写入文件

## 使用方法

```bash
# 添加任务
python main.py add "完成软件工程实验1"

# 查看所有任务
python main.py list

# 完成任务（1 为任务编号）
python main.py done 1
```

## 数据结构

每个任务包含以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int | 任务编号，唯一且递增 |
| title | string | 任务标题 |
| status | string | todo（待办）或 done（已完成） |
| created_at | string | 创建时间，格式 YYYY-MM-DD HH:MM:SS |

## 实验反思

**问题：为什么"会写 Python"还不等于"会做软件"？**

会写 Python 意味着掌握了语法、数据结构和算法，能把一个明确的想法翻译成代码。但做软件远不止写代码——

首先，软件始于需求。从同学访谈中我发现，用户说的往往是模糊的（"我怕忘记交作业"），而不是精确的（"我需要一个支持 JSON 持久化的命令行任务管理器"）。把真实问题转化为可实现、可验收的功能描述，是写代码之前最关键的一步。

其次，软件必须可交付、可运行。一个只能在自己的 IDE 里跑通的脚本不是软件。需要明确入口、参数处理、错误提示、数据持久化——这些不是 Python 语法问题，而是工程意识。

再者，软件需要可维护。当前 main.py 不过 70 行，但已经在 load/save/add/list/done 几个职责之间耦合。若不加设计地继续堆功能，很快就会变成没人敢改的"脚本怪"。这也是实验二模块化重构的动机。

总结：写 Python 是"造砖"，做软件是"造房子"——前者只需要语法，后者需要需求分析、设计、测试、交付和维护的完整链条。
